# v1.1.0-rc.1 — Multi-flock support

This release candidate adds support for running **multiple flocks** on one machine — one coordinator per target repo, sharing a dashboard.

## Highlights

- **Multiple flocks per install.** Each flock has its own `.env`, SQLite queue, GitHub Apps, chat tokens, and pm2 process at `~/.flockbots/instances/<slug>/`. Shared resources (Supabase project, dashboard login, webhook relay, Claude rate-limit budget) live at the root.
- **`flockbots init` picker.** Re-running with at least one flock configured offers: reconfigure one, add a new flock, or edit shared settings. New flocks auto-inherit Supabase / dashboard login / relay from siblings — only chat tokens, GitHub Apps, and the target repo are per-flock.
- **`flockbots instances`** — list configured flocks with target repo, chat provider, and pm2 status (online / stopped / pm2 down).
- **`flockbots remove [-i <slug>]`** — archive a single flock without touching the others. Stops pm2, marks the Supabase row archived, removes the instance dir, cleans the target repo's worktrees, and prints a revocation checklist that hedges if another flock still uses the same GitHub App.
- **`-i <slug>` flag** on per-flock commands (`task add`, `kg build`, `dashboard deploy`, `webhook deploy`, `doctor`, `remove`). Auto-resolved when only one flock exists.
- **Dashboard instance switcher** in the top bar scopes pipeline / events / usage / office view to the selected flock via Supabase realtime filters; escalations stay cross-flock.
- **GitHub App reuse.** When adding flock #2+, the wizard JWT-pre-flights existing FlockBots Agent / Reviewer apps and offers to install them on the new repo instead of creating a fresh pair.
- **pm2 added as a required prereq** with an interactive install offer at the end of `flockbots init`.
- **Empty-repo handling** — the wizard auto-falls to single-branch mode when the target repo has no commits yet (two-branch needs a primary branch to branch staging from).

## Breaking changes

The Supabase schema is **incompatible with v1.0.x**:

- Every coordinator-written table now has `instance_id NOT NULL REFERENCES flockbots_instances(id)`.
- Composite primary keys: `flockbots_tasks (instance_id, id)`, `flockbots_escalations (instance_id, id)`, `flockbots_system_health (instance_id, key)`.
- New `flockbots_instances` registry table tracks each coordinator's `display_name`, `target_repo`, `chat_provider`, `last_seen_at`, `archived_at`.
- Realtime subscriptions now filter on `instance_id=eq.<slug>` for per-instance scoping.

The on-disk layout is also incompatible: `data/`, `tasks/`, `logs/`, `keys/`, `.env` move from `~/.flockbots/` root to `~/.flockbots/instances/<slug>/`.

## Migration

**Fresh install only.** No in-place migration from v1.0.x is provided. To upgrade:

1. Stand up a new Supabase project for the v1.1 schema (free tier is fine).
2. `flockbots uninstall` your v1.0.x install (revoke chat bots / GitHub Apps as instructed).
3. `curl -fsSL https://raw.githubusercontent.com/pushan-hinduja/flockbots/main/install.sh | bash`
4. `flockbots init` against the new Supabase project. Re-create the GitHub Apps via the wizard.
5. If you have multiple repos, `flockbots init` again for each one to add additional flocks.

## Full changelog

- Multi-instance Supabase schema (`supabase/migrations/consolidated.sql` rewritten — fresh install required).
- Per-instance on-disk layout: every coordinator file moves from `~/.flockbots/` root to `~/.flockbots/instances/<slug>/`.
- New CLI commands: `flockbots instances`, `flockbots remove`.
- New `-i <slug>` flag on `task add`, `kg build`, `dashboard deploy`, `webhook deploy`, `doctor`, `remove`.
- New wizard flow: top-level instance picker (`coordinator/src/cli/wizard-instances.ts`), Linear team+project picker (`wizard-linear.ts`), GitHub App reuse with JWT pre-flight.
- Per-instance webhook routing: relay's `[slug].ts` Edge route stamps `instance_id` per request.
- Dashboard `InstanceContext` + `InstanceSwitcher` + every per-instance hook scoped via realtime filters.
- pm2 prereq + `offerPm2Install` at end of `flockbots init`.
- Supabase URL/keys + dashboard login auto-reused from siblings on instance #2+.
- Empty-repo single-branch fallback in the deploy-strategy step.
- Stale-flash fix in dashboard hooks (clear before re-fetch on instance change).
- Coordinator startup registers the instance in `flockbots_instances` (`upsertInstance`) before any sync write; periodic `heartbeatInstance` for dashboard liveness dots.
- Shared rate-limit budget at `~/.flockbots/rate-limit-state.json` (atomic-rename writes) so all flocks share one Claude account.

## Verification status

This is a **release candidate**. Local sandbox testing was blocked by a pm2 quirk with paths containing spaces (production `~/.flockbots/` has no spaces). The schema migration, full pipeline run, dashboard switcher with hyphenated slugs, and webhook relay route have not yet been smoked end-to-end against a real Supabase + Vercel + chat provider. Promote to `v1.1.0` once those pass on a clean install.
