# v1.1.0 — Multi-flock support

The headline feature of this release: FlockBots can now run **multiple flocks** on one machine — one coordinator per target repo, sharing a single dashboard.

## Highlights

### Multi-flock architecture

- One install, many repos. Each flock keeps its own `.env`, SQLite queue, GitHub Apps, chat tokens, and pm2 process at `~/.flockbots/instances/<slug>/`.
- Shared resources (Supabase project, dashboard login, webhook relay, Claude rate-limit budget) live at the flock root and are auto-reused across flocks.
- New CLI commands:
  - `flockbots instances` — list every configured flock with target repo, chat provider, and pm2 status.
  - `flockbots remove [-i <slug>]` — archive a single flock without touching the others. Stops pm2, marks the Supabase row archived, removes the instance dir, cleans the target repo's worktrees, and prints a revocation checklist that hedges if other flocks share the same GitHub App.
  - `-i <slug>` flag on `task add`, `kg build`, `dashboard deploy`, `webhook deploy`, and `doctor`. Auto-resolved when only one flock exists.

### Wizard updates

- `flockbots init` detects existing flocks and offers a picker: reconfigure one, add a new flock, or edit shared settings.
- Adding a second flock auto-inherits the Supabase project, dashboard login, and webhook relay from sibling flocks — only chat tokens, GitHub Apps, and target repo need to be picked per flock.
- **GitHub App reuse:** the wizard JWT-pre-flights existing FlockBots Agent / Reviewer apps and offers to install them on a new repo instead of creating a fresh pair.
- **Auto-restart on reconfigure:** when an existing flock that's currently running under pm2 has its `.env` rewritten by reconfigure, the wizard offers to restart that flock so the new env vars take effect immediately.
- **Empty repos work out of the box:** the wizard auto-falls to single-branch mode when the target repo has no commits, and the coordinator seeds the staging branch on first task creation.

### Dashboard

- **Instance switcher** in the top chrome scopes every panel (pipeline, events, usage, office view, sub-agents) to the selected flock via Supabase realtime filters.
- **Escalations stay cross-flock** — they page the operator regardless of which flock is currently in focus, prefixed with the flock the escalation came from.
- **Pixel office routing:** agents now reroute around stuck peers instead of waiting in place; refreshing the dashboard or switching instances snaps active agents to their desks instead of walking from a random spawn.
- **Merged-stage breakdown** gains a `QA FAILED` segment (red) alongside passed / in-QA / awaiting / awaiting-human, so post-merge regressions stop being invisible.
- Custom dark-themed dropdown for the instance picker; UTC clock no longer wraps to a second line; heartbeat bar restored to its full layout.

### Vercel deploy rewrite

- `flockbots dashboard deploy` and `flockbots webhook deploy` now drive the **`vercel` CLI** directly against your local `~/.flockbots/dashboard/` and `~/.flockbots/webhook-relay/` directories. The previous `vercel.com/new/clone?...` URL flow forked the entire monorepo into the user's GitHub and was prone to hanging in Vercel's import UI.
- Three clack-branded prompts (scope picker, link-to-existing confirm, project name) replace Vercel CLI's six native questions. Project conflicts in your scope are detected via the Vercel API and surfaced before linking.
- **`flockbots upgrade` automatically redeploys** any linked Vercel project after rebuilding the coordinator — dashboard, relay, and coordinator advance in lockstep instead of drifting apart.
- Auth is browser-based on first run (`vercel login`) and the token is shared across every flock + survives reinstalls. `VERCEL_TOKEN` env var bypasses the prompt for headless / CI use.

### Setup polish

- **pm2 added as a required prereq**, with an interactive install offer at the end of `flockbots init`.
- **Knowledge-graph script** reads from per-instance `.env` (was hardcoded to the v1.0 root layout, so the wizard's kg step silently failed).
- **Telegram poll-error noise filter:** single-cycle network blips appear once at info level in `out.log` and never reach `error.log` unless the streak hits 5 consecutive failures, with a recovery message when polling resumes.
- **Knowledge-graph build errors** now surface the last 30 lines of script output instead of just "exited with code N".

## Breaking changes

**Supabase schema is incompatible with v1.0.x:**

- Every coordinator-written table now has `instance_id NOT NULL REFERENCES flockbots_instances(id)`.
- Composite primary keys: `flockbots_tasks (instance_id, id)`, `flockbots_escalations (instance_id, id)`, `flockbots_system_health (instance_id, key)`.
- New `flockbots_instances` registry table tracks each coordinator's `display_name`, `target_repo`, `chat_provider`, `last_seen_at`, `archived_at`.
- Realtime subscriptions filter on `instance_id=eq.<slug>` for per-instance scoping.

**On-disk layout** also incompatible: `data/`, `tasks/`, `logs/`, `keys/`, `.env` all move from `~/.flockbots/` root to `~/.flockbots/instances/<slug>/`.

## Migrating from v1.0.x

Fresh install only. No in-place migration is provided.

1. Stand up a new Supabase project (free tier is fine).
2. `flockbots uninstall` your v1.0.x install — revoke chat bots / GitHub Apps as instructed by the cleanup checklist (or keep GitHub Apps for v1.1's reuse flow).
3. `curl -fsSL https://raw.githubusercontent.com/pushan-hinduja/flockbots/main/install.sh | bash`
4. `flockbots init` against the new Supabase project. Re-create or reuse GitHub Apps via the wizard.
5. If you have multiple repos, run `flockbots init` again for each — the wizard auto-inherits shared values from your first flock.
